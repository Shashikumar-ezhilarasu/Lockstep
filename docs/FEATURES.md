# Implemented Features Matrix

This document provides a tabular overview of all features currently implemented in the Lockstep distributed job scheduler, summarizing what each feature does and how it is technically realized in the codebase.

| Feature | Description (What it does) | Technical Realization (How it is implemented) |
| :--- | :--- | :--- |
| **Multi-tenant Isolation** | Ensures users can only access or mutate resources (projects, queues, jobs, DLQ) belonging to their organization. | REST API endpoints decode JWTs via Fastify `preHandler` hooks and verify membership in the `org_members` table before running DB queries. |
| **Project & Queue Partitioning** | Groups resources logically: Organization $\rightarrow$ Projects $\rightarrow$ Queues $\rightarrow$ Jobs. | Tables `projects` and `queues` use cascade constraints. Queue definitions hold priority, concurrency limits, and default retry policy links. |
| **Queue Pausing / Resuming** | Allows queue administrators to instantly suspend or resume job executions globally. | API updates the `queues.status` to `paused` / `active`. The worker's claiming query checks `queues.status = 'active'` directly, causing paused queues to be ignored. |
| **Immediate Jobs** | Enqueues jobs for immediate processing by the next available worker. | API inserts jobs with `status = 'queued'` and `scheduled_at = now()`. The worker polls and claims these immediately. |
| **Delayed Jobs** | Executes a task after a specified delay duration. | API accepts `delay_ms` and inserts jobs with `status = 'scheduled'` and `scheduled_at = now() + delay_ms`. The scheduler sweeper updates them to `queued` once the delay expires. |
| **Scheduled One-time Jobs** | Triggers a job execution at an explicit future datetime. | API accepts `scheduled_at` timestamp and writes it directly to the job record. The scheduler daemon wakes the job up when due. |
| **Recurring Cron Jobs** | Dispatches new job runs repeatedly according to a cron expression. | Cron definitions are stored in `scheduled_jobs`. The scheduler daemon evaluates expressions using `cron-parser` to schedule execution and update `next_run_at`. |
| **Batch Jobs** | Groups multiple task payloads together under a single batch identifier. | API enqueues multiple items sharing a single `batch_id`. The dashboard aggregates batch completion status using a custom SQL aggregation endpoint (`/batches/:batchId`). |
| **Workflow Dependencies** | Enforces execution order (DAGs) so a job only runs after its dependencies succeed. | Claim query in `claim.ts` uses a `NOT EXISTS` subquery against `job_dependencies` to filter out jobs whose parent tasks are not `completed`. |
| **Atomic Job Claiming** | Guarantees that distributed workers do not pull or execute the same job twice. | Worker claims jobs using `FOR UPDATE OF jobs SKIP LOCKED`, allowing workers to immediately lock available rows and skip already-locked items. |
| **Global Concurrency Limits** | Restricts the maximum number of concurrent running tasks per queue across all workers. | Computed inside a `SERIALIZABLE` transaction query that calculates `concurrency_limit - active_jobs`. Aborted transaction conflicts (error `40001`) are automatically retried with randomized backoff. |
| **Job Retries & Backoff** | Automatically schedules re-attempts for failed tasks with fixed, linear, or exponential delay backoffs. | FSM transitions failed jobs back to `scheduled` and computes the next run timestamp based on configured `retry_policies`. |
| **Dead Letter Queue (DLQ)** | Quarantines jobs that have exhausted all retry attempts, preventing execution loops. | Workers transition terminal jobs to `dead_letter` and write to the `dead_letter_queue` table. API supports requeuing (resetting attempts to 0) or deletion. |
| **Execution Logging** | Records complete worker output logs and durations for observability. | Workers write history records to the `job_executions` table and step-by-step logs to the `job_logs` table, exposing them via `/jobs/:jobId/logs` REST APIs. |
| **Worker Heartbeats** | Monitors worker health and detects crashes. | Workers report to the `worker_heartbeats` table every 10 seconds. The scheduler service sweeps for stale workers (no heartbeat $>30$s) to flag them `offline`. |
| **Stale Job Recovery** | Recovers jobs that were left running on workers that crashed. | The scheduler service identifies jobs marked `claimed` or `running` by a stale worker, closes their execution, and resets their status to `queued`. |
| **Graceful Shutdown** | Drains active tasks when worker processes receive termination signals. | Workers intercept `SIGTERM`/`SIGINT`, update their status to `draining`, stop claiming new tasks, and wait up to 30 seconds for active executions to complete. |
| **WebSocket Live Updates** | Re-renders dashboard components instantly when database changes occur without REST API polling. | Frontend dashboard connects to Supabase Realtime WebSockets to broadcast database WAL (Write-Ahead Log) updates directly to React tables. |
